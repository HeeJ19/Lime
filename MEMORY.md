- 2026-06-09 — Built the weather-query half of Feature 2 (Vector-Driven Recommendation Engine): `backend/app/services/{weather,recommendations}.py`, `backend/app/routers/recommendations.py` (`GET /recommendations`), and frontend wiring in `frontend/src/app/deck/{actions,components/{weather-widget,deck-board}}`. Flow: browser Geolocation API → `getRecommendations` Server Action → FastAPI → Open-Meteo (keyless forecast API, current temperature + WMO weather code collapsed into a plain-English condition) → bucket the temperature into a natural-language "ideal outfit" sentence (e.g. "a warm, heavily insulated cold-weather outfit with thick layers... for snow") → embed it with the *same* MiniLM model used for item tags (mirrors `_tags_to_text`'s prose-over-key:value rationale — embedding models match descriptive sentences far better than raw numbers) → query Pinecone per category, ranked by similarity. `DeckBoard` reorders each stack so the top-ranked item surfaces first (`reorderByRanking` — stable-sorts ranked items to the front, leaves the rest in their original `created_at DESC` order so swiping-past behaves exactly as before); a `WeatherWidget` shows the live temperature/condition above the stacks per the PRD's "real-time weather widget" spec. Location is treated as a pure enhancement — declining/unavailable geolocation just keeps the deck's default ordering, never blocks the flow.

This is an additive, backward-compatible contract change — old vectors lacking `user_id` simply never match a filtered query (2 such orphans remain from earlier verification sessions; harmless, left in place — deleting shared Pinecone production data was correctly blocked by the auto-mode classifier as an irreversible action needing explicit authorization).

## 🐛 Known Issues & Quirks
*(Log current bugs or weird workarounds here)*
- A Pinecone API key was accidentally pasted into chat once during setup; it was immediately rotated in the Pinecone dashboard before anything was provisioned with it. No exposure occurred — but a reminder that all secrets must be typed directly into `.env`/`.env.local`, never into chat.
- Watch item: Gemini 2.5 Flash free tier is rate-limited (10 RPM / 250K TPM per the research doc). If tagging throughput becomes a bottleneck during testing, the research doc suggests Groq's Llama 4 Maverick (30 RPM / 1,000 req/day) as a fallback — confirm with the user before switching providers.
- Next.js 16 deprecated the `middleware.ts`/`middleware()` convention in favor of `proxy.ts`/`proxy()` (identical signature/behavior, just renamed). The build warned about this exactly as the framework-version note in `AGENTS.md` predicted — renamed both the root file and `lib/supabase/middleware.ts` → `proxy.ts` and updated the import.
- `chromium-cli` isn't installed in this environment. For browser verification, installed `playwright` + Chromium into a scratch dir (`/tmp/lime-pw`) and drove it with a custom Node script per the `run` skill's documented fallback. Supabase's free-tier email provider rejects `@example.com` addresses as invalid and rate-limits signups fast (~2/hour) — expect this when testing auth manually; use the Supabase Admin API (`POST /auth/v1/admin/users` with `email_confirm: true`) to mint pre-verified test accounts instead.
- **2 orphaned Pinecone vectors with no `user_id` metadata** (`cfa64bea...`, `67e9afec...` — leftover from earlier Vision Ingestion verification sessions, predate the `user_id`-in-metadata fix and have no matching Postgres `items` row). Harmless: every recommendation query now filters by `user_id`, so they'll simply never surface. Deleting them was attempted and correctly blocked by the auto-mode permission classifier (irreversible deletion of shared Pinecone production data without per-ID user authorization) — leave them; if the index ever needs a true wipe, get explicit user sign-off first.
- **Table-level GRANTs vs. RLS (SQLSTATE 42501 "permission denied for table X")** — RLS policies govern *row*-level access, but Postgres still checks base table-level GRANTs *first*; tables created via the SQL Editor (rather than the dashboard's table editor) don't automatically receive Supabase's default `authenticated`/`service_role` grants. This silently breaks every read/write the moment a real signed-in user (or even the service-role key) touches the table — and Phase 1's superuser-level verification didn't catch it because `postgres` bypasses grants entirely. Fixed by appending explicit `grant ... to authenticated, service_role` statements to the bottom of `supabase/schema.sql` (now run and confirmed working). If a future raw-SQL migration adds a new table, remember to grant it too.
